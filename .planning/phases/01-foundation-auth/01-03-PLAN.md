    The SQL has THREE sections — copy each in order:

    **Section 1 — Non-superuser role (idempotent):**
    ```sql
    DO $$
    BEGIN
      IF NOT EXISTS (SELECT 1 FROM pg_roles WHERE rolname = 'foray_app') THEN
        CREATE ROLE foray_app LOGIN PASSWORD 'CHANGE_ME_VIA_ENV';
      END IF;
    END$$;

    GRANT USAGE ON SCHEMA public TO foray_app;
    GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO foray_app;
    GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA public TO foray_app;

    ALTER DEFAULT PRIVILEGES IN SCHEMA public
      GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO foray_app;
    ALTER DEFAULT PRIVILEGES IN SCHEMA public
      GRANT USAGE, SELECT ON SEQUENCES TO foray_app;
    ```

    **Section 2 — ENABLE + FORCE ROW LEVEL SECURITY on every tenant-scoped table:**
    Apply BOTH `ALTER TABLE ... ENABLE` and `ALTER TABLE ... FORCE` for each of:
    `users`, `companies`, `applications`, `stages`, `events`, `emails`, `recruiters`, `application_recruiters`, `documents`

    Without `FORCE`, the table owner bypasses policies (Pitfall #2). NEVER skip a `FORCE` line.

    **Section 3 — tenant_isolation policies:**
    - `users`: `USING (id = current_setting('app.user_id', true)::int) WITH CHECK (...)` — User has no user_id column; its id IS the user
    - `companies`, `applications`, `events`, `emails`, `recruiters`: `USING (user_id = current_setting('app.user_id', true)::int) WITH CHECK (...)`
    - `stages`, `application_recruiters`, `documents`: parent-scope via `USING (application_id IN (SELECT id FROM applications WHERE user_id = current_setting('app.user_id', true)::int)) WITH CHECK (...)`

    Critical invariants (all from RESEARCH.md §1 "Critical gotchas avoided"):
    1. ALWAYS use `current_setting('app.user_id', true)` — the `, true` (missing_ok) returns NULL on unset, policy denies. Without it: throws.
    2. ALWAYS pair `ENABLE` with `FORCE`. Without FORCE: owner bypass. Without ENABLE: no RLS at all.
    3. The `users` policy uses `id =`, NOT `user_id =`. User has no user_id column.
    4. `::int` cast — schema uses `Int` for ids, not UUID.

    Also add a Prisma migration name comment at the top:

    ```ts
    import { execSync } from 'node:child_process'

    export default async function globalSetup() {
      const container = await new PostgreSqlContainer('postgres:16')
        .withDatabase('foray_test')
        .withUsername('foray_owner')   // migration role (NOT foray_app)
        .withPassword('test')
        .start()

      const ownerUrl = container.getConnectionUri()

      // Apply schema as the owner role.
      execSync('pnpm prisma migrate deploy', {
        env: { ...process.env, DATABASE_URL: ownerUrl },
        stdio: 'inherit',
      })

      // Seed two users (alice = id 1, bob = id 2) for cross-tenant tests.
      // Inline SQL keeps the seed scope tight; production seed.ts stays single-user.
      const { Client } = await import('pg')
      const client = new Client({ connectionString: ownerUrl })
      await client.connect()
      await client.query(`
        INSERT INTO users (email, name, created_at, updated_at)
          VALUES ('alice@test', 'Alice', NOW(), NOW()),
                 ('bob@test',   'Bob',   NOW(), NOW())
          ON CONFLICT (email) DO NOTHING;
      `)
      // Set the foray_app password to a known test value so connections work.
      await client.query(`ALTER ROLE foray_app PASSWORD 'test_app_pw';`)
      await client.end()

      // Switch the test process to use foray_app for the rest of the run.
      const appUrl = ownerUrl.replace('foray_owner:test', 'foray_app:test_app_pw')
      process.env.DATABASE_URL = appUrl

      return async () => { await container.stop() }
    }
    ```
    Note on the seed: alice + bob get inserted via raw SQL as the OWNER role (FORCE RLS doesn't apply pre-policy-enable... but it does after Section 2 of the migration runs. Counter-intuitively, the seed must come AFTER `migrate deploy`. The SQL above uses INSERT directly as the owner — but FORCE RLS will block even the owner. Solution: the OWNER role connects, but we use `SET LOCAL app.user_id = ...` per-row, OR we connect as superuser to bypass.) **Use this safer variant instead:**
    ```ts
    // Disable RLS temporarily for seed insertion (we are the owner; FORCE RLS would block).
    // After seed, the rest of the run uses foray_app (non-superuser, FORCE RLS active).
    await client.query(`SET row_security = off;`)
    // ... INSERT statements ...
    ```
    `SET row_security = off` only works when the role has `BYPASSRLS` OR when using the `superuser` mode the Testcontainers `postgres:16` image runs as. Testcontainers' default role (`foray_owner` here) is created as superuser by the Postgres image initialization — so `SET row_security = off` works for the seed phase. Verify by reading `node_modules/@testcontainers/postgresql/dist/postgresql-container.d.ts` after install if in doubt.
