  <behavior>
    - generateApiToken() returns a 32-byte hex string (64 chars)
    - hashToken(token) returns a bcrypt hash of the token
    - verifyApiToken(userId, token) returns true if token matches stored hash, false otherwise
    - revokeApiToken(userId) sets the hash column to null
    - Token is only shown once at generation time; DB stores only the hash
  </behavior>
  <action>
    1. Add to `prisma/schema.prisma` on the User model:
       ```prisma
       extensionApiTokenHash String? @map("extension_api_token_hash")
       ```
       Run `pnpm prisma migrate dev --name add-extension-api-token` to create migration.

    2. Create `src/core/auth/api-token.ts` with:
       - `generateApiToken(): string` — uses `crypto.randomBytes(32).toString('hex')`
       - `hashToken(token: string): Promise<string>` — uses bcrypt hash (bcryptjs is already a transitive dep via iron-session; if not available, use `crypto.createHash('sha256')` with a salt stored alongside). Actually, per PRINCIPLES.md §Security, use bcrypt or argon2. Check if bcryptjs is in deps; if not, use Node.js crypto with `scrypt` (built-in, no new dep).
       - `verifyApiToken(userId: UserId, token: string): Promise<boolean>` — fetches `extensionApiTokenHash` from User via `prisma` (this is auth-layer, goes through `core/db` not `tenantDb` since it's verifying identity), compares hash
       - `revokeApiToken(userId: UserId): Promise<void>` — sets `extensionApiTokenHash` to null
       - `storeApiTokenHash(userId: UserId, hash: string): Promise<void>` — updates user record with hash

       Use Node.js built-in `crypto.scrypt` for hashing (no new dependency). Format: `scrypt:<salt>:<hash>` stored as a single string column.

    3. Create `src/core/auth/api-token.test.ts` with unit tests:

       - hashToken produces consistent hashes for same input
       - verifyApiToken returns true for correct token, false for wrong token
       - verifyApiToken returns false for non-existent user
       - revokeApiToken clears the hash

    4. Follow project conventions: kebab-case file names, `ok/err` from neverthrow, branded UserId type.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| extension -> capture API | Untrusted client (extension) sends requests with Bearer token |
| settings page -> token generation | Authenticated user generates sensitive credential |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-01 | Spoofing | capture route | mitigate | Verify Bearer token hash against DB; token shown only once at generation |
| T-13-02 | Information Disclosure | token generation | mitigate | Token displayed once, then only hash stored; use crypto.scrypt (timing-safe comparison) |
| T-13-03 | Tampering | capture route | mitigate | Existing URL validation + ATS domain rejection already in place; CORS headers limit surface |
| T-13-04 | Denial of Service | capture route | accept | Single-user local deployment; existing MAX_BODY_BYTES guard sufficient |
</threat_model>
