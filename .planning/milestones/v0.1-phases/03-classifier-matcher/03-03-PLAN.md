Build the matcher slice — a read-only DB function that maps `{userId, gmailThreadId, fromDomain}` to `applicationId | null` via the LOCKED 4-step tiebreak from CONTEXT §Area 5.

Plan 03-03 is **independent of Plans 03-01 and 03-02** (matcher does not call classifier; classifier does not call matcher; they are PEER slices per AGENTS.md module boundaries). It can therefore run parallel to 03-01 in Wave 1.

Purpose: Phase 4's `inbox.pollOnce` calls `matchEmail` per email AFTER ingest, BEFORE classify. The matcher is read-only — it does NOT write `email.applicationId`. Phase 4's act-stage handles the write.

The single critical anti-pattern this slice prevents: the ATS-domain false-positive (Pitfall #5). Step 2 short-circuits this.

```typescript
export async function matchEmail({ userId, gmailThreadId, fromDomain }: MatchEmailInput):
  Promise<Result<{ applicationId: ApplicationId | null }, AppError>> {
  const tdb = tenantDb(userId)
  // 1. thread continuity
  const threadEmail = await tdb.email.findFirst({
    where: { gmailThreadId, applicationId: { not: null } },
    orderBy: { receivedAt: 'desc' },
  })
  if (threadEmail?.applicationId) return ok({ applicationId: ApplicationId(threadEmail.applicationId) })

  // 2. ATS skip BEFORE domain match
  if (isAtsDomain(fromDomain)) return ok({ applicationId: null })

  // 3. domain match
  const company = await tdb.company.findFirst({
    where: { domain: fromDomain },
    include: { applications: { orderBy: { appliedAt: 'desc' }, take: 1 } },
  })
  if (company?.applications[0]) return ok({ applicationId: ApplicationId(company.applications[0].id) })

  // 4. unmatched
  return ok({ applicationId: null })
}
```

```typescript
import 'server-only'
import { ok, err, fromPromise, type Result } from 'neverthrow'
import { tenantDb } from '@/core/db'
import { isAtsDomain } from '@/core/domains/ats-domains'
import { ApplicationId, type UserId } from '@/core/types/ids'
import { errors, type AppError } from '@/core/errors'
import { matchEmailInputSchema, type MatchEmailInput, type MatchEmailOutput } from './schema'

export async function matchEmail(
  rawInput: MatchEmailInput
): Promise<Result<MatchEmailOutput, AppError>> {
  const parsed = matchEmailInputSchema.safeParse(rawInput)
  if (!parsed.success) return err(errors.validation(parsed.error.issues))
  const { userId, gmailThreadId, fromDomain } = parsed.data as MatchEmailInput  // userId already branded

  const tdb = tenantDb(userId as UserId)

  return fromPromise(
    (async (): Promise<MatchEmailOutput> => {
      // 1. Thread continuity — most-recent email on the same thread that's already linked
      const threadEmail = await tdb.email.findFirst({
        where: { gmailThreadId, applicationId: { not: null } },
        orderBy: { receivedAt: 'desc' },
        select: { applicationId: true },
      })
      if (threadEmail?.applicationId) {
        return { applicationId: ApplicationId(threadEmail.applicationId) }
      }

      // 2. ATS-domain skip — BEFORE step 3 (Pitfall #5: ATS domains owned by recruiters/applicants ≠ company)
      if (isAtsDomain(fromDomain)) {
        return { applicationId: null }
      }

      // 3. Sender-domain match — find a company whose domain equals fromDomain, take most-recent application
      const company = await tdb.company.findFirst({
        where: { domain: fromDomain },
        include: {
          applications: {
            orderBy: { appliedAt: 'desc' },
            take: 1,
            select: { id: true },
          },
        },
      })
      if (company && company.applications.length > 0) {
        const firstApp = company.applications[0]
        if (firstApp) return { applicationId: ApplicationId(firstApp.id) }
      }

      // 4. Unmatched
      return { applicationId: null }
    })(),
    (cause) => errors.db(cause),
  )
}
```

CRITICAL invariants:
- ATS skip in step 2 fires BEFORE step 3. NOT after — must short-circuit so the domain query never runs for ATS domains.
- Step 1's `applicationId: { not: null }` is essential — we want emails ALREADY linked to an application, not raw thread membership.
- Step 1's `orderBy: { receivedAt: 'desc' }` handles the edge case of multiple thread-linked emails on the same thread → return the most recent (CONTEXT §"Claude's Discretion": "if multiple emails on the same thread are linked to DIFFERENT applications — return the most recent").
- Step 3's `take: 1` + `orderBy: { appliedAt: 'desc' }` returns the most-recent application for the matched company. CONTEXT §"Claude's Discretion": single match (most-recent), not all. Limitation is documented.
- The returned `null` case is `ok(null)`, NOT `err(...)`. Unmatched is a normal outcome (the email goes to review queue in Phase 4).
- Service does NOT mutate `email.applicationId`. Phase 4's act-stage handles the write after both match + classify succeed.
