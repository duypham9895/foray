<objective>
Build the public `classifyEmail` service — the function Phase 4 will import. It composes Plan 03-01's three pure utilities (rules, thresholds-via-imported-cutoff, budget) with a thin Anthropic SDK wrapper.

The composition order is the trust contract:
1. Rules-first (cheap, deterministic, no I/O)
2. Short-circuit on confident rule match (≥0.85) OR explicit unmatched (no rule fired)
3. Budget gate BEFORE the SDK call (fail-closed control, not monitoring)
4. SDK call with structured tool output (rejects unstructured responses)
5. Cost recording AFTER successful SDK response (not before — we don't charge for failures)

Purpose: Phase 4's `inbox.pollOnce` calls `classifyEmail({subject, bodyExcerpt})` per email. The function is PURE w.r.t. the database (no Prisma access here; Phase 4's pipeline does the email row reads/writes). Idempotency on `email.classifiedBy` is enforced by Phase 4 BEFORE calling — Phase 3 just provides the function.

**service.ts (CLASS-01 + CLASS-02 + CLASS-04 composition):**
```typescript
import 'server-only'
import { ok, err, type Result } from 'neverthrow'
import { errors, type AppError } from '@/core/errors'
import type { EmailClassification } from '@/generated/prisma/client'
import { classifyByRules } from './rules'
import { classifyByLlm, MODEL } from './llm'
import { checkBudget, appendCostEntry, hashEmailContent } from './budget'
import { classifyEmailInputSchema } from './schema'

export type ClassifyEmailInput = { subject: string; bodyExcerpt: string }
export type ClassifyEmailOutput = {
  label: EmailClassification
  confidence: number
  classifiedBy: 'rules' | 'llm'
}

export async function classifyEmail(
  input: ClassifyEmailInput
): Promise<Result<ClassifyEmailOutput, AppError>>
```

Algorithm (LOCKED in CONTEXT §Area 2):
```
1. parsed = classifyEmailInputSchema.safeParse(input)
   if !parsed.success → return err(errors.validation(parsed.error.issues))

2. rules = classifyByRules(parsed.data)

3. // Short-circuit: rules confident, accept
   if (rules.label === 'unmatched') {
     return ok({ label: 'unmatched', confidence: 0, classifiedBy: 'rules' })
   }
   if (rules.confidence >= 0.85) {
     return ok({ label: rules.label, confidence: rules.confidence, classifiedBy: 'rules' })
   }

4. // Rules matched but weakly — escalate to LLM
   const budget = await checkBudget()
   if (budget.isErr()) return err(budget.error)  // RateLimited

5. const llm = await classifyByLlm(parsed.data)
   if (llm.isErr()) return err(llm.error)

6. // Record cost (best-effort; don't fail classification if logging fails)
   const emailHash = hashEmailContent(parsed.data.subject, parsed.data.bodyExcerpt)
   await appendCostEntry({
     inputTokens: llm.value.inputTokens,
     outputTokens: llm.value.outputTokens,
     model: MODEL,
     emailHash,
   })  // intentional: ignore the Result; logging failure is not a classification failure

7. return ok({ label: llm.value.label, confidence: llm.value.confidence, classifiedBy: 'llm' })
```

CRITICAL invariants:
- Step 3 (short-circuit) MUST run BEFORE budget check — rule-confident cases never touch the budget gate.
- Step 4 (budget check) MUST run BEFORE the SDK call — control, not monitoring.
- Step 6 (cost record) runs only on LLM SUCCESS — failures don't get logged as cost (Anthropic doesn't bill failed requests).
- Step 6's appendCostEntry result is intentionally NOT awaited-checked. Per Karpathy §1.2: classification SUCCESS shouldn't fail because logging failed. The trade-off is documented: a write-failure means we under-count cost. Acceptable for Lean (single-user). Add a `// HACK(duy, 2026-05-09): cost-log write failure ignored — classification succeeded; logging is best-effort. Standard milestone: route this to a separate alerting channel.` comment.
