    ## Status: Accepted
    ## Date: 2026-05-09 (or current date — use the date Plan 05 runs)

    ## Context

    Two decisions Phase 2 made that future phases will be tempted to revisit. Both are cross-cutting (per ROADMAP.md §"Cross-Cutting Concerns" — trust trio).

    Sub-section A — "Status regression":
    Phase 4's `inbox/act` stage will call `applyAutoStatusChange` for every email the classifier labels with `confidence ≥ threshold`. Without a guard, a single ambiguous email could un-reject a foray (e.g., a "did you receive our previous email?" reply on a thread previously classified as rejection could classify as `interview_invite` and overwrite a rejection). This is the failure mode Pitfalls #4 (status regressions) calls out.

    Two valid approaches considered:
    1. Guard at the controller (Phase 4 inbox/act) — keep the service permissive, let the orchestrator decide.
    2. Guard at the service (Phase 2 applications/service.ts) — service refuses regressions; controller routes to review queue on Conflict.

    Sub-section B — "Auto-update event visual treatment":
    Auto-applied status changes need to be visually distinct from manual edits in the timeline (per APP-02 + DESIGN.md). A user must be able to scan the timeline and instantly tell which entries the system did vs which entries they did, especially when reviewing what the classifier acted on.

    DESIGN.md constraints: no decorative icons; colors used sparingly; rejection rendered in muted gray (NEVER red); status colors anchored to the canonical_status palette (cyan-600 is the `screening` hue, signalling "in-progress / gentle attention").

    Three valid visual treatments considered:
    1. Background-only tint (no rail, no label).
    2. Icon + text (e.g., a robot icon + "Auto").
    3. Tinted background + colored left rail + text label, no icon.

    ## Decision

    A. **Status regression is enforced at the service layer.**
       - `src/features/applications/status-transitions.ts` exports `isStatusRegression(prev, next)` as the single source of truth.
       - `src/features/applications/service.ts:applyAutoStatusChange` calls `isStatusRegression` after reading the current status, before writing the new one — all inside one `withRls` transaction. Rejections return `err({_tag: 'Conflict', reason: 'STATUS_REGRESSION_REQUIRES_REVIEW'})` and the transaction rolls back.
       - `applyManualStatusChange` does NOT call `isStatusRegression` — manual changes by the user are intentional and bypass the guard.
       - Terminal-status semantics (`rejected` and `withdrawn` both at rank 5):
         - Any non-terminal → terminal is FORWARD (closing out is normal lifecycle progress).
         - Any terminal → non-terminal is REGRESSION (un-rejecting must be a deliberate human undo via `undoStatusChange`, not an automated reclassification).
         - Terminal ↔ terminal (rejected ↔ withdrawn) is NO-MOVEMENT — neither forward nor regression.

    B. **Auto-update events render with cyan-tinted row + 2px cyan-600 left rail + "Auto-updated from email" text label.**
       - Background: `bg-cyan-50` light, `dark:bg-cyan-950/30` dark.
       - Left rail: `border-l-2 border-cyan-600` (the `screening` hue per DESIGN.md — signals "system in progress" without alarm).
       - Label: `<span className="text-sm text-stone-500">Auto-updated from email</span>` — text only, no icon (per DESIGN.md §"Less icons, more humanity").
       - Implementation: `src/features/applications/components/timeline.tsx`.
       - Manual events (status_changed, stage_added, stage_completed, note_added, status_undone) get a plain row, no rail, no label.
       - Undone auto-events (`undoneAt IS NOT NULL`) get `text-stone-400 line-through` to show the action was reversed; the original visual treatment is preserved as a strikethrough so the audit trail stays visible.

    ## Consequences

    ### Positive
    - Phase 4's inbox/act stage is a thin orchestrator: try `applyAutoStatusChange`, on Conflict route to review queue. No regression logic in Phase 4 — it is locked here.
    - The timeline visual is consistent across Phase 2 (manual + classifier-fixture-driven seed) and Phase 4 (real Gmail-driven auto-events) without further coordination.
    - Terminal-status rule prevents the exact bug from Pitfalls #4 ("interview_invite" on a thread post-rejection).
    - Cyan-600 anchors auto-update events in the `screening` hue family — semantically aligned with "in progress, gentle attention".

    ### Negative
    - Service-layer regression block means Phase 4 must distinguish Validation/NotFound errors (which surface as user-visible) from Conflict errors (which silently route to review queue). Mitigation: structured `AppError._tag` already supports this.
    - The visual treatment requires Tailwind classes spread across timeline.tsx — if a future redesign moves to a different design system, these classes need to migrate. Mitigation: the classes are documented here so the redesign can find them via `git grep "border-cyan-600"` + this ADR reference.

    ## When we'd reconsider

    - **Phase 4 reveals a class of email signal where regression is correct** (e.g., a reopened-position email from the same recruiter). At that point: introduce a `forceRegression` flag on `applyAutoStatusChange` requiring explicit caller intent; do NOT remove the default block.
    - **The classifier becomes confident enough that the regression block is too conservative** (e.g., a hypothetical Phase 6 with multi-signal corroboration). At that point: add an additional confidence-tier flag.
    - **DESIGN.md adopts a different status palette.** Update this ADR + supersede with ADR-0XXX and migrate timeline.tsx classes in one PR.
